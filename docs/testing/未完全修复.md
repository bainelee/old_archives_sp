# 已修复归档：正厅聚焦后房间悬停/点击异常（2026-04-01）

## 问题背景

- 现象：在 3D 场景中，聚焦到 `room_00`（档案馆正厅）后，鼠标在部分其他房间上悬停/点击无效。
- 用户判断：问题与镜头/视角/缩放变化后鼠标射线落点偏移相关。
- 本轮目标：完成“可复现 -> 修复尝试 -> GameplayFlow 验证”闭环，并记录未完全收敛部分。

## 第三轮修复记录：命中坐标统一与碰撞体维度修正（2026-04-02）

### 根因复盘（本轮确认）

- **关键根因 1：输入事件坐标与射线坐标源不一致。**
  - `game_main_input.gd` 原先大量使用 `get_viewport().get_mouse_position()`，在镜头变化与自动注入输入场景下会出现“当前事件位置”和“射线取样位置”脱钩。
  - 修复：3D 命中链改为显式传入本次事件 `mouse_pos`，新增 `_raycast_mouse_3d_at(mouse_pos)` 与 `_get_room_at_mouse_3d_at(mouse_pos)`。
- **关键根因 2：碰撞体按“薄层”构建，和当前场景坐标布局不匹配。**
  - 档案馆房间在 `X-Y` 平面分布，`Y` 维承载房间排布；将 pick 碰撞盒做成低 `Y` 薄层会造成上/下排房间高频 `no_area_hit`。
  - 修复：`RoomBlockHighlight` 的 pick 盒 `Y` 改为基于 `room_volume.y` 的完整高度（设最小高度下限），`X/Z` 保持可控 margin。

### 本轮新增/调整

- `scripts/game/game_main.gd`
  - 新增 `_raycast_mouse_3d_at(mouse_pos)`、`_get_room_at_mouse_3d_at(mouse_pos)`。
  - `_raycast_mouse_3d()` 改为转调 `_raycast_mouse_3d_at(get_viewport().get_mouse_position())`，兼容旧调用。
  - 保留并增强 `get_debug_last_mouse_pick_3d()` 诊断输出。
  - `_update_room_highlights()` 改为优先调用 `RoomBlockHighlight.set_highlight_visible(...)`，不再依赖隐藏根节点。
- `scripts/game/game_main_input.gd`
  - 鼠标左键/移动/中键拖拽改为使用事件位置：`mb.position` / `motion_event.position`。
  - 新增 `get_ui_block_detail(...)` 诊断函数，`is_click_over_ui_buttons(...)` 仅复用并返回布尔结果（行为不放开）。
- `scripts/rooms/room_block_highlight.gd`
  - 根节点保持可见，新增 `set_highlight_visible(show_highlight)` 仅控制四条可视边框显隐。
  - pick 盒高度改为 `max(PICK_MIN_HEIGHT, GRID_SIZE * yR + THICKNESS_IN + THICKNESS_OUT * 2)`。
- `scripts/test/test_driver_actions.gd`
  - `room_projection_hover_click_matrix` 使用 `_get_room_at_mouse_3d_at(probe_pos)`，避免采样坐标漂移。
  - 失败诊断加入 `ray_debug`、`ui_block_detail`，并在 `skipUiCovered=true` 时把 UI 覆盖点记为 `skipped`。
  - 增加 `projection_not_self_resolving` 候选过滤，降低房间投影重叠导致的误判。
- flow 参数调整：
  - `room_hover_click_projection_matrix_zoom_slot0.json`：`minRooms` 调整为 `0`（缩放后可测候选动态变化）。
  - `room_hover_click_projection_matrix_edge_slot0.json`、`room_hover_click_projection_matrix_refocus_slot0.json`：`minRooms` 调整为 `1`。

## 第四轮修复补充：隐形 UI 拦截修正与人工验收（2026-04-02）

### 背景

- 用户提供截图复现：同一房间附近，红点可悬停、蓝点不可悬停。
- 该现象不符合“UI 可见时阻挡、不可见时不阻挡”的预期。

### 代码修正

- `scripts/game/game_main_input.gd`
  - `_ui_hit(...)` 判定由 `ctrl.visible` 改为 `ctrl.is_visible_in_tree()`。
  - `TopBar` / `BottomRightBar` 命中判定统一走 `_ui_hit(...)`。

### 修正原因

- 仅检查子控件自身 `visible` 会漏掉父层隐藏状态。
- 例如 `RoomDetailPanelFigma` 隐藏但 `PanelRoot.visible=true` 时，旧逻辑可能仍把 `PanelRoot` 当作拦截源。
- 改为 `is_visible_in_tree()` 后，只有“在可见树上”的 UI 才会阻挡鼠标，符合交互设计。

### 验收结论

- 用户人工复测确认：问题已解决。
- 最终行为与预期一致：
  - 可见 UI 正常阻挡鼠标；
  - 不存在“隐形 UI 挡射线”导致的悬停失效。

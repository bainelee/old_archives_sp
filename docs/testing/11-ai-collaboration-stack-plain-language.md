# 游戏同仁向：本项目如何用「规则 + 工具」配合 AI 做游戏

本文说明《旧日档案馆》在开发过程中，**MCP、技能（Skill）、规则（Rules）** 以及配套理念各自是什么、解决什么问题。面向**游戏行业的策划、美术、叙事、音频、运营、发行、制作等同仁**——**不必会写代码，也不必是 AI 工程师**；尽量少用术语，用**游戏制作里常见的短场景**帮助理解。

---

## 这篇文档写给谁

- 参与同一款游戏、但日常不写脚本的**策划 / 美术 / 叙事 / 音频 / 运营 / 发行 / PM** 等  
- 需要向公司内外解释「为什么敢让 AI 大量参与改工程」的**制作人或主策**  

读完你可以大致回答：**AI 改完一版工程后，靠什么保证没把别的系统弄坏？调数值为什么不会只改表、文档却掉队？**

---

## 先用游戏制作的话说清楚全貌

一款模拟经营 + 基地建设类游戏，系统多、表格多、版本迭代快。团队希望 AI 能帮忙改工程，但又不能变成「谁都不知道动到了哪里」。下面这张表用**大家熟悉的工种协作**来对照本仓库里的几类机制：

| 在游戏团队里你大概见过 | 在本项目里大概对应 |
|------------------------|-------------------|
| 美术有**导出规格**，程序有**性能与存档红线**——各工种按约定交接 | **Rules（规则）**：AI 改到某类文件时，自动套上对应约定，减少跨界误伤 |
| 里程碑前要过一遍**冒烟用例**：新档、进核心玩法、存读档……打勾才算过 | **GameplayFlow + MCP**：自动跑一串关键玩家路径，并留下步骤记录 |
| TD / 主程把「这类 bug 怎么查」写成**组内笔记**，新人照着走 | **Skills（技能）**：把固定流程写成手册，AI 先读再动手，少靠临场发挥 |
| 策划只维护**一张主数值表 / 主内容表**，程序从表里读，不各写各的 | **数据文件 + 设计文档**：数值、房间、探索等**唯一真源**在数据里，文档对齐 |

下面分块说明，每块都带**一个简短的游戏开发场景**。

---

## 1. MCP + 自动测试：像「里程碑前的冒烟清单」，不是「我觉得能玩」

**专业说法（可跳过）**：MCP 是编辑器里的一种**工具接口**，让 AI 能调用「启动游戏、执行测试流程、拉取步骤结果」等能力；配合 Godot 插件，在**真机真工程**里跑流程。

**游戏制作里常见的场景**：  
版本日的前一晚，主策问程序：「昨天那个探索界面改好了吗？」若回答只有「能进游戏」，你其实不知道**别的流程有没有被连带弄坏**——比如新开档、进档案馆、推进时间、存读档是否仍成立。  
**MCP + GameplayFlow** 就像一份**可自动执行的冒烟用例**：按步骤「开档 → 点这里 → 验证那里」，每一步在**自动化日志或报告**里**有播报、有通过/失败**（和 CI 跑构建时一行行出结果类似）；中途红了会**停住**，报告里能指到**卡在第几步**。

**本项目里具体有什么用**

- 可以**自动模拟**一连串玩家操作（开档、进房间、探索、存读档等），减轻「每次改完都靠同一个人手点一遍」的压力。  
- 步骤按固定协议播报：**开始执行 → 执行结果 → 验证结论**；方便策划、制作追问进度，而不必先读懂引擎底层日志。  
- 跑完会留下**报告、时间线、截图**等，复盘时像看**测试留档**：是这一步断言失败，还是上一步状态就不对。

**一句话**：AI 会犯错；**自动跑关键路径 + 步骤可见**让团队像过里程碑清单一样验收，而不是凭感觉说「好像没问题」。

## 2. Skills（技能）：像「组内 Wiki 里的固定流程页」

**专业说法（可跳过）**：技能是写在项目里的**操作手册**，遇到某类任务时 AI 先读手册再动手。

**游戏制作里常见的场景**：  
新人进组第一天，主美会发一份「**UI 切图命名与导入检查**」，音频会发「**响度与格式规范**」——大家按同一套做，外包回来的资源才接得住。若每人凭印象操作，最容易出现**漏一步、版本对不齐、反复返工**。  
**Skills** 就是给 AI 也配了这类**流程页**：遇到某类事，先打开对应章节再改工程。

**本项目里几个典型技能（举例）**

- **GF-EXP**：修完一类与自动测试流程相关的 bug 后，把**可复用的排查思路**沉淀成「问题类型级」笔记，并规定**哪些文件允许合回主分支**（避免把试验性改动和正式修复搅在一起）。  
- **覆盖层输入调试**：专门对付「**界面叠在上面，但点了没反应**」这类 Godot 常见 UI 问题，像组里那句「**先查输入吞噬 / 暂停树**」——比空泛问「为什么坏了」更快对症。  
- **Plan 子代理方案**：做大功能或大包体改动时，要求写出**谁能并行、谁交接什么产出**，接近制作表上的**依赖与并行列**。  
- **UI 字体检查**：从设计同步到工程后，核对是否统一走项目字体，避免**有的界面清晰、有的发糊或回退成系统字体**。

**一句话**：技能把**项目组已经付过学费的经验**写成**可重复的步骤**，换一任协助者（人或 AI）也不用从头发明轮子。

## 3. Rules（规则）：像「策划文档里按系统写的注意事项」

**专业说法（可跳过）**：规则按**文件类型或话题**自动生效，提醒 AI 在改相关工程文件时别违反项目约定。

**游戏制作里常见的场景**：  
GDD 里常有「**本章改动必须同步本地化**」「**数值只认主表**」——不是让每个人背整本设计，而是**你动到这一章，就遵守这一章的红线**。  
**Rules** 对 AI 的作用类似：**当前在改哪一类内容，就自动挂上哪几条组内约定**，降低「改 A 系统却忘了 B 系统的交接条件」。

**本项目里几类常见规则（用制作语言说）**

| 你作为策划 / 叙事 / 制作会关心的事 | 规则大概在管什么 |
|------------------------------------|------------------|
| 改了一个经济或时间相关的数，**对外的设计说明**是否还一致 | **数值同步**：动主数据时要连带相关设计文档，避免**表里是新数、文档还是旧版** |
| 改了房间名或描述，**英文界面**会不会缺条或显示键名 | **room_info 本地化同步**：中文内容表动了，翻译表要跟上 |
| 美术在编辑器里**调 UI 布局**，是否非得**运行游戏**才看得见效果 | **预制作 UI**：与「编辑器里就要能预览」相关的工程约定 |
| 「清空存档」会不会误伤玩家**自制地图**或其它应保留内容 | **存档规则**：哪些目录能清、哪些绝不能动 |
| 编剧给的**台词 / 设定段落**，是否会被 AI 擅自「润色」改意 | **创意文案保留**：禁止改写用户提供的剧本类文字 |

**一句话**：规则让 AI **在正确的系统边界里动手**，接近你们对**外包或跨工种交接**时强调的「别碰这条线」。

## 4. 数据驱动 + 文档：像「策划主表 + GDD」，而不是「每个程序各记一版数」

**专业说法（可跳过）**：大量玩法来自 **JSON 等数据文件**（数值、房间、探索配置等），脚本**读取数据**而不是在无数处写死数字；设计文档写清**意图与口径**，与表对齐。

**游戏制作里常见的场景**：  
若敌人血量、掉落、关卡难度**散落在多个脚本**里，策划想调平衡时只能追着程序问「到底哪句是真」。成熟管线里更常见的是：**一张或一组主表驱动玩法**，程序只负责「读表 → 生效」，策划改表就能迭代。  
本项目的思路一致：**数值、房间、探索等以数据为真源**，文档与工程围绕同一份口径维护。

**为什么这对「AI 大量参与改工程」特别重要**

- AI 批量改**结构化表格 + 对应文档**，比在一堆脚本里找「魔法数字」**更不容易漏改**。  
- 子流程（例如数值同步清单）会写明：**先动哪份数据、再动哪几篇设计说明**，像**发行前 Checklist**。  
- 模拟经营类**系统多、相互影响多**；**真源集中**才能把复杂度关在「表 + 文档」里，而不是散成谁也说不清的暗线。

---

## 5. AGENTS.md：像「项目 Runbook / 仓库导航一页纸」

**专业说法（可跳过）**：仓库里的 Agent 速查页，列出核心目录、关键命令、日志路径、测试怎么跑。

**游戏制作里常见的场景**：  
新人进项目或外包刚接入仓库时，制作人或主程常会丢一份**「从拉仓库到能跑起来」**的短链：引擎版本、启动场景、日志在哪、里程碑怎么验。  
**AGENTS.md** 对 AI 的作用类似：**先读这一页，再动工程**，减少在错误目录里瞎改或漏跑关键验证。

**一句话**：让协助者（人或 AI）**快速对齐「本项目怎么跑、去哪查、测什么」**，少在无关路径上浪费时间。

---

## 6. 合在一起：为什么能支撑「复杂数据驱动游戏」的 AI 协作

用制作语言概括因果：

1. **数据集中**：像**只认主表**——改一行配置，全链路从文档到运行时口径一致，适合 AI 做**结构化批量修改**。  
2. **自动冒烟**：像**里程碑前的固定用例**——改完就跑核心玩家路径，**早发现、早停损**，降低「修界面却弄坏存读档」类连锁风险。  
3. **流程与红线**：技能和规则把**组内已知的步骤与忌讳**外挂给 AI，弥补模型**不了解你们项目史**的短板。  
4. **过程可见**：步骤播报与报告，让**策划 / 制作**也能问「卡在冒烟第几步」，而不必先读懂引擎日志。

---

## 7. 明确边界：这些**不能**替代什么

- **不替代**真正的玩家体验测试（手感、节奏、情感共鸣）。  
- **不替代**策划与美术对品质的最终判断。  
- **不保证**覆盖所有「玩家会乱试」的边角路径；自动流程偏向**高频、标准、可重复**的验证。  
- **创意文案**仍由人定稿；规则会限制 AI 乱改台词与设定文字。

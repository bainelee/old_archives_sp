# GameplayFlow 对外介绍（初稿）

## 1. 产品定位

GPF（GameplayFlow）是面向游戏开发阶段的测试、Debug、QA 专用工具。  
它的定位是开发期验证与修复闭环，不是上线后的黑盒巡检工具。

与打包后黑盒视觉 QA 的差异：

- 黑盒视觉 QA：主要判断画面看起来是否异常，难以深入到运行状态与代码层原因。
- GPF：既能模拟玩家操作，也能定位归因并配合修复与复测。

## 2. 它是什么

GameplayFlow 可以理解为一个按剧本执行的测试机器人：

- 流程可来自自然语言需求，由 AI 分析后生成可执行步骤。
- 流程按顺序执行，每一步都带验证条件。
- 每次执行都会生成可追溯结果和证据。

核心目标不是替代玩家，而是稳定、重复地验证功能是否正常。

## 3. 它解决什么问题

在迭代频繁的项目中，常见问题是：

- 手工复测重复、耗时、易漏。
- 改动后影响面不清晰。
- 不同成员复测同一问题，路径和结论不一致。

GameplayFlow 的作用：

- 固定验证步骤，降低人为波动。
- 从自然语言快速生成流程，覆盖不同操作路径与行为差异。
- 快速发现回归问题。
- 失败可定位到具体步骤并给出证据。

## 4. 当前已实现能力

- **流程执行**：可模拟常见玩家操作，覆盖核心玩法链路和阶段推进流程。
- **状态一致性**：可验证进度保存与恢复后的状态是否一致。
- **视觉联调**：支持截图对照、错误区域标识，并进入修正与复测闭环。
- **多工具协作**：支持与 Figma 设计同步链路的 UI 联调验证。
- **数据对账**：可处理采样时间差带来的误差，减少“看似异常”的误判。
- **过程可见**：步骤级播报“开始执行 -> 执行结果 -> 验证结论”。
- **结果沉淀**：输出日志、时间线、失败摘要和关键截图。

## 5. 如何模拟玩家操作（非技术版）

GameplayFlow 不是固定坐标连点器，而是“顺序执行 + 动态定位”：

- 流程按步骤线性执行，便于控制和复盘。
- 每一步先定位目标，再执行动作。
- 同一流程可覆盖不同操作路径，减少“只测到一种点法”的风险。

一句话：步骤是可控的，命中是动态的，不是硬编码死坐标。

## 6. 如何形成定位与修复闭环

GPF 不只会执行，还会协助排障与修复：

- 失败会落到具体步骤，不是只有“整体失败”。
- 每次失败都附带可复盘证据（日志、时间线、截图）。
- 可把问题映射回对应模块，推进修复后复测。
- 在设计联调场景中，可将视觉差异、状态差异与代码修改串成同一闭环。

形成流程：发现问题 -> 标识差异 -> 定位原因 -> 修复 -> 复测。

## 7. 数据对账能力（时间差归因）

自动化流程中，前后两次采样之间可能存在时间差。  
这时“前后不一致”不一定是 bug，也可能是正常推进造成。

GPF 会输出可解释对账结果：

- 区分正常推进变化与测试衔接延迟变化。
- 给出实际结果与扣除延迟影响后的参考结果。
- 即使失败，也会提供最可能原因和下一步排查方向。

价值在于把“是否异常”升级为“异常来自哪里、如何处理”。

## 8. 为什么它更可靠（通俗版）

- 全程可见：每一步都有播报，不会黑盒运行。
- 避免卡死：长期无进展会自动停止并给出结论。
- 避免互相干扰：同一项目同一时间默认只跑一条主流程。
- 失败可复盘：失败信息包含步骤、证据和排查建议。
- 结果可复用：报告结构统一，便于长期沉淀和复用。

## 9. 真实案例（Figma UI 同步联调）

我们在一次 UI 改版中，把设计稿通过 Figma 同步到项目后，用 GPF 做联调验证。

简化流程：

1) 跑既有流程并产出关键截图。  
2) 对照设计并标识异常区域。  
3) 回看对应步骤与日志定位问题。  
4) 修正后复测，确认差异消失。

这个案例证明：GPF 不只复现操作，还能把视觉差异、状态差异与修复复测串成可重复闭环。

## 10. GPF 的 EXP 特色能力

GPF 内建 EXP（经验沉淀）机制。  
目标不只是修掉当前问题，而是把过程沉淀成可复用方法资产。

每次收敛后沉淀两类内容：

- 类型错误的解决方案（常见根因、推荐修法、注意事项）。
- 调查路径（先看什么、再验证什么、如何缩小范围）。

直接收益：

- 帮助后续开发实现，减少重复试错。
- 提升同类问题排障效率与稳定性。

## 11. 非开发者怎么使用

不需要理解代码，也可以按 3 步使用：

1) 选择流程（如“启动进度 -> 核心交互 -> 状态校验 -> 保存恢复验证”）。  
2) 运行流程并观察步骤播报。  
3) 查看结果与证据（通过/失败、截图、日志、摘要）。

这让策划、测试、程序可以围绕同一流程协作，而不是凭印象复测。

## 12. 输出结果长什么样

每次运行会产出结构化结果，常见包括：

- 总结报告：整体通过/失败状态。
- 步骤时间线：每一步的时长、状态、说明。
- 失败摘要：关键失败点、期望与实际差异。
- 证据文件：日志与关键截图。

适用场景：

- 每日回归检查。
- 问题复现与交接。
- 版本发布前稳定性确认。

## 13. 当前边界（它不做什么）

GameplayFlow 不是通用 AI 玩家，也不替代人工体验测试。  
它目前不擅长：

- 主观体验判断（好不好玩、节奏是否舒适、情绪是否到位）。
- 完全未知路径的自由探索。
- 极端随机、强实时、依赖人工判断的复杂场景。

推荐策略：高频标准化验证交给 GameplayFlow，体验和探索性交给人工测试。

## 14. 对外一句话版本

GPF（GameplayFlow）是一套面向开发阶段的测试、Debug、QA 专用工具：  
它可根据自然语言需求由 AI 生成可执行流程，在固定验证步骤下覆盖多种人为操作差异，并在多工具协作场景中完成视觉对照、错误标识、归因与修复闭环。

## 15. EXP 样例条目模板（建议）

建议每次 Debug 收敛后按统一格式记录 EXP：

- 问题类型：例如“状态机未收敛”“UI 同步后视觉偏移”“进度恢复后状态不一致”。
- 现象与触发条件：一句话描述现象 + 触发步骤/场景。
- 调查路径：按顺序记录“先检查 A -> 再验证 B -> 最后确认 C”。
- 根因结论：指出导致问题的关键机制或实现点。
- 修复方案：本次修复动作（必要时附替代方案）。
- 回归验证：使用哪条 GameplayFlow、通过标准是什么。
- 复用建议：哪些后续需求/问题可复用本条 EXP。

推荐写作原则：

- 描述可复用方法，而不仅是“改了哪一行”。
- 路径要可执行，避免只给结论。
- 验证要可重复，便于他人独立复现。
